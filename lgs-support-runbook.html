01 System Overview & Criticality

System NameLoading Guidance System (LGS)
PurposeGuides warehouse staff on optimal truck loading sequences and configurations across global fulfillment sites
UsersWarehouse operations staff (internal employees), warehouse supervisors
DeploymentGCP — GKE / Cloud Run · Global multi-region
Scope5–20 cities globally, multiple time zones
Operating Hours06:00–22:00 local time per warehouse site (effectively near-24/7 globally)
CriticalityTier 2 — Business Critical
Manual Fallback✅ Available — staff can execute manual loading procedures during system degradation
ℹ️ Criticality Rationale

LGS is classified Tier 2 (Business Critical) rather than Tier 1 (Mission Critical) because manual fallback procedures exist. However, prolonged outages (>4h) during warehouse peak hours significantly impact operational efficiency and throughput across multiple sites.

02 Incident Severity Levels

All incidents must be assigned a severity level at the time of triage. The following definitions apply specifically to LGS during its initial launch phase.

Severity Name Definition Example
P1 · Critical System-Wide Outage LGS is completely unavailable across 3+ warehouse sites simultaneously. Manual fallback activated. Major operational impact. GKE cluster crash; Cloud Run service 100% error rate globally; database unreachable
P2 · High Site-Level Outage LGS is unavailable or severely degraded for 1–2 specific sites. Workers at those sites cannot use the system. Workaround in use. Single region pod crash; config error causing one site to fail; authentication broken for one site
P3 · Medium Feature Degradation System is accessible but a specific loading guidance feature is broken or inaccurate. Core functionality still works. Workaround available. Incorrect loading sequence suggestion; PDF report failing to generate; slow response (>10s)
P4 · Low Minor / Cosmetic Minor UI issue, incorrect text/label, non-blocking bug. No operational impact. Wrong label on button; minor display glitch; feature request misrouted as bug
⚠️ Initial Launch Phase — Severity Inflation Rule

During the first 90 days post-launch, treat all first-occurrence bugs at one severity higher than normal classification. Unknown failure modes are more likely and early signal collection is critical. Downgrade after root cause confirmed.

04 Incident Reporting Process

For Warehouse Staff — How to Report

✅ Primary Channel: ServiceNow

All incidents must be reported via ServiceNow. Do not report via email or informal channels — this ensures proper SLA tracking.

Step-by-Step: Creating a ServiceNow Ticket

1
Navigate to ServiceNow Portal
Login to your company's ServiceNow instance. Select "Report an Issue" or "Create Incident".
2
Fill in Required Fields
Complete all mandatory fields:
Affected System: Loading Guidance System (LGS)
Site / Warehouse: [Your city and warehouse code]
Impact: How many staff are affected? Can work continue?
Description: What were you doing? What happened? What was expected?
First Occurrence: Exact time and date (local timezone)
Screenshot/Video: Attach evidence if possible
3
Select Initial Priority
Use the severity definitions in Section 2 to select the appropriate priority. When in doubt, select higher severity — L1 will adjust during triage.
4
Submit & Save Ticket Number
Note your INC-XXXXXXX ticket number. You will receive automated updates via email when the status changes.
5
Activate Manual Fallback (if needed)
If the system is unavailable, activate the manual loading procedure (refer to your local SOP). Do not wait for system recovery to continue operations.

05 Triage Process (IKEA-Aligned)

LGS follows IKEA's standard 3-tier triage model aligned with the CDHF/ITIL framework. All incidents in ServiceNow are routed through this pipeline.

L1
L1 — First Response & Initial Triage
Owner: Global Support Desk (follows-the-sun)
Goal: Acknowledge within SLA, assess initial severity, apply known fixes from runbook
Actions:
① Confirm the issue is reproducible
② Check GCP Cloud Run / GKE health dashboard
③ Check Grafana/Loki for error spike in lgs-api service
④ Apply runbook fix if known pattern (see Section 8)
⑤ If unresolved within 30 min → escalate to L2

Tools: ServiceNow · GCP Console · Grafana Loki · Slack/Teams alert channel
L2
L2 — Engineering On-Call
Owner: LGS Engineering Team (on-call rotation)
Goal: Root cause identification, patch or rollback decision
Actions:
① Pull incident from ServiceNow + Jira
② Search historical incidents for similar patterns
③ Correlate Grafana logs: logcli query '{service_name="lgs-api"}' |= "ERROR"
④ Identify root cause (code / config / infra / data)
⑤ Apply fix or rollback deployment
⑥ Confirm resolution with L1 / reporter
⑦ Document root cause in ServiceNow ticket

Decision Gate: If root cause requires architectural change or vendor involvement → escalate to L3
L3
L3 — Specialist / Vendor Escalation
Owner: Senior Engineer / GCP Support / External Vendor
Goal: Resolve complex infrastructure-level or vendor-dependent issues
When to escalate to L3:
• GCP platform incident (GKE node failure, Cloud Run quota exceeded)
• Data corruption requiring DBA involvement
• Security incident (potential breach)
• Issue affecting ALL sites simultaneously with unknown root cause

L3 Contacts: See Escalation Matrix (Section 6)

⚠️ Initial Launch Phase

During the first 90 days post-launch, double the normal on-call staffing. New systems have a high rate of unexpected failures and configuration issues. Two engineers per zone should be on-call (primary + secondary) to reduce response time and avoid single-engineer burnout.

On-Call Responsibilities

  • Monitor #lgs-alerts Slack/Teams channel during your on-call window
  • Respond to PagerDuty pages within 15 minutes for P1, 30 minutes for P2
  • Execute runbooks for known patterns before escalating
  • Document all actions taken in the ServiceNow ticket
  • Hand off open incidents with status summary at zone transition
  • Complete an Incident Report for all P1 and P2 incidents within 24h of resolution

08 Common Runbooks (Initial Set)

📝 Living Document

These runbooks are pre-populated based on common GKE/Cloud Run failure patterns. L2 engineers must add new entries after every P1/P2 incident resolution. Target: 80% of incidents resolved via runbook within 3 months of launch.

RB-001: All Sites Unreachable

1. Check GCP Cloud Run service status:
   GCP Console → Cloud Run → lgs-api → check for 5xx error rate
   
2. Check GKE cluster health (if applicable):
   kubectl get pods -n lgs-production
   kubectl describe pod [failing-pod] -n lgs-production

3. Check Loki for error patterns:
   logcli query '{service_name="lgs-api"} |= "ERROR"' --limit=50
   
4. If OOM/CPU: Scale up replicas:
   kubectl scale deployment lgs-api --replicas=5 -n lgs-production
   
5. If bad deployment: Rollback:
   kubectl rollout undo deployment/lgs-api -n lgs-production
   
6. Verify recovery: test from 2 different sites before closing

RB-004: Slow Response (>10 seconds)

1. Check Cloud SQL query performance:
   GCP Console → Cloud SQL → lgs-db → Query Insights
   
2. Check for missing index on loading_plan table (common on new data):
   Run: EXPLAIN ANALYZE SELECT * FROM loading_plans WHERE site_id = ?
   
3. Check GKE pod memory/CPU:
   kubectl top pods -n lgs-production
   
4. If database connections exhausted:
   Restart connection pool: kubectl rollout restart deployment/lgs-api
   
5. If ETL data backlog causing full-table scan:
   Contact data engineering team to pause and replay ETL job

10 Support Staffing Estimate

Based on: 5–20 city global coverage · 06:00–22:00 warehouse operating hours · 3 time zones · Initial launch phase · GKE/Cloud Run deployment.

🟦
L1 Global Support (Follow-the-Sun) 2.0 FTE
One L1 support agent per time zone shift covering APAC / EMEA / AMER warehouse hours. Responsibilities: ServiceNow queue monitoring, ticket triage, runbook execution, status updates to reporters.

Breakdown: 3 time zone shifts × ~8h overlap each = 2 FTE equivalent (with some shared shift coverage). Shared with existing IKEA global support desk is preferred model — dedicated headcount if volume exceeds 5 tickets/day.
🟠
L2 On-Call Engineering 1.5 FTE
Engineering on-call rotation across 3 time zones. One primary + one secondary per zone during initial launch phase (first 90 days). After stabilization, reduce to one primary only.

Breakdown: 3 time zones × 1 engineer primary + 1 secondary (initial) = effectively 1.5 FTE of dedicated on-call capacity (rotation across team of 4–6 engineers, with on-call weeks shared).

Tools needed: PagerDuty integration with GCP Monitoring, Loki/Grafana access, kubectl access to production cluster.
🔴
L3 Engineering Specialist (Shared) 0.25 FTE
Senior engineer or tech lead available for P1 escalation. Not a dedicated role — shared responsibility within the engineering team. ~5–10% of working time expected on L3 incidents in initial months.
📊
Incident Ops / SRE (if team >10 cities) 0.5 FTE
Optional role as system scales. Owns runbook maintenance, SLA reporting, post-incident reviews, and proactive monitoring improvements. Can be a shared SRE role across multiple systems.

11 SLA Review & Continuous Improvement

Monthly SLA Review Checklist

  • ☐ Calculate actual resolution times vs. SLA targets for each severity
  • ☐ Review tickets breaching SLA — identify root causes (process gap? staffing gap? technical gap?)
  • ☐ Update runbooks with new failure patterns discovered in the month
  • ☐ Review on-call burnout metrics — adjust rotation if needed
  • ☐ Check if severity re-classification was appropriate (retrospective)
  • ☐ Share summary with warehouse operations stakeholders

90-Day Post-Launch Review

📅 Mandatory Milestone

At day 90, conduct a formal review to: (1) assess actual incident volume vs. estimates, (2) right-size staffing, (3) consider removing "initial launch phase" severity inflation rule if system is stable, (4) update SLA targets based on observed system performance.
