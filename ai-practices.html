🔧 Tool Calling 替代"让模型直接输出结构化 JSON"

让模型专注语义生成，工具负责结构正确性，显著降低幻觉风险

⚡ 优先级 P1 难度：中
❌ 问卷系统踩坑

让模型一次性输出完整 question JSON → 幻觉概率高、结构漂移、偶发乱码不可解析

✅ 测试平台应用

AI 生成测试用例时，不直接输出 JUnit/Postman JSON，而是调用工具，工具负责组装

💡 核心原则：模型关注"语义生成"，工具负责"结构正确性"。两者职责分离后，幻觉风险大幅下降。
架构设计
# ❌ 错误做法：让模型直接输出完整 JSON
prompt = "生成一个完整的 JUnit XML 格式测试用例，包含所有字段..."
# 结果：格式漂移、字段缺失、偶发乱码

# ✅ 正确做法：模型只生成参数，工具负责组装
tools = [
    {
        "name": "generateTestCase",
        "description": "生成标准测试用例",
        "parameters": {
            "title": "string",       # 模型生成
            "steps": "array",       # 模型生成
            "expected": "string",    # 模型生成
            "priority": "enum",      # 强类型约束
            "type": "enum"          # 强类型约束
        }
    }
]

# 工具内部：统一组装为标准 Kiwi TCMS / JUnit 格式
def generateTestCase(title, steps, expected, priority, type):
    return {
        "summary": title,
        "text": "\n".join(steps),
        "expected_duration": 0,
        "priority": PRIORITY_MAP[priority],  # 枚举校验
        "category": TYPE_MAP[type],
        "notes": expected
    }

同时支持"参考式生成"：传入已有测试用例或上传 API Spec 文件（JSON/YAML）作为参考，生成质量更稳。

💓 SSE 心跳保活（长耗时 Tool Calling 必做）

防止网关在 AI 思考期间误判连接空闲而断开，是流式 AI 场景的通用解法

🔴 优先级 P0（必做） 难度：低
❌ 问卷系统踩坑

Tool Calling 组装参数时无输出，Kong 网关 30 秒超时主动断连，前端报错流程中断

✅ 测试平台场景

批量生成测试用例（复杂 Story）时，AI 思考阶段可能超过 30 秒，必须加心跳

⚠️ 如果你们也在用 Kong 网关，必须同时打开：konghq.com/response-buffering: "false"，否则即使加了心跳，流也会被"攒包"，用户看到的是"假流式"。
python — Cloud Function 心跳实现
import asyncio
from sse_starlette.sse import EventSourceResponse

async def generate_test_cases_stream(story_id: str):
    """流式生成测试用例，含心跳保活"""

    async def event_generator():
        # 启动心跳任务
        heartbeat_task = asyncio.create_task(heartbeat())

        try:
            # AI 生成（可能有较长 pending 阶段）
            async for chunk in call_llm_with_tools(story_id):
                yield {"event": "data", "data": chunk}
        finally:
            heartbeat_task.cancel()
            yield {"event": "done", "data": "[DONE]"}

    async def heartbeat():
        while True:
            await asyncio.sleep(5)          # 每 5 秒一次
            yield {"event": "heartbeat", "data": "ping"}

    return EventSourceResponse(event_generator())

# 前端忽略心跳事件
# eventSource.addEventListener('heartbeat', () => {}) // 不做任何处理

🔢 Tool Calling 限额，防止死循环

工程化硬约束优于提示词软约束，超限返回结构化错误让模型自然收敛

🔴 优先级 P0（必做） 难度：低
💡 关键设计原则

超限时不抛异常，而是返回结构化错误作为"正常工具返回值"喂回模型。抛异常会直接中断 SSE 流，用户体验变成"生成到一半突然断了"。

python — 请求级计数器实现
from contextvars import ContextVar
from functools import wraps

# 每次请求独立计数，请求结束自动重置
_tool_call_counts: ContextVar[dict] = ContextVar('tool_call_counts', default={})

TOOL_LIMITS = {
    "generateTestCase":  20,  # 单次最多生成 20 个
    "fetchApiSpec":       2,   # API Spec 最多查 2 次
    "searchJiraStory":    3,   # Jira 查询最多 3 次
}

def with_call_limit(tool_name: str):
    def decorator(func):
        @wraps(func)
        def wrapper(*args, **kwargs):
            counts = _tool_call_counts.get({})
            current = counts.get(tool_name, 0)
            limit = TOOL_LIMITS.get(tool_name, 10)

            if current >= limit:
                # 不抛异常，返回结构化提示让模型收敛
                return {
                    "error": "TOOL_CALL_LIMIT_EXCEEDED",
                    "tool": tool_name,
                    "limit": limit,
                    "attempt": current + 1,
                    "message": f"该工具最多调用 {limit} 次，请基于已有信息完成输出。"
                }

            counts[tool_name] = current + 1
            _tool_call_counts.set(counts)
            return func(*args, **kwargs)
        return wrapper
    return decorator

# 使用方式
@with_call_limit("fetchApiSpec")
def fetchApiSpec(endpoint: str):
    return get_api_spec_from_swagger(endpoint)

📊 大批量数据分层处理（MapReduce 策略）

结构化数据前置计算，非结构化文本分治摘要，避免 Token 超限崩溃

🟠 优先级 P1 难度：中
测试平台应用场景

AI 分析大量测试结果时（如 1000+ 条失败日志），不要直接塞给模型，分层处理速度更快、成本更低、不崩溃。

架构设计 — 三层分治
"""
测试结果 AI 分析链路

层次 1：结构化数据前置计算（不走 AI）
  pass/fail 统计、覆盖率、各团队对比 → 代码计算
  速度快、成本 ¥0、结果 100% 准确

层次 2：文本数据 MapReduce（> 500 条失败日志时启用）
  Map:    把失败日志按 100 条一批 → Qwen-flash 分批摘要（快且便宜）
  Reduce: 所有分批摘要 → Qwen-plus 聚合成总结

层次 3：全局分析报告
  层次1结构化指标 + 层次2文本摘要 → Sonnet/Qwen-plus 生成最终报告
"""

async def analyze_test_results(run_id: str):
    # 层次 1：代码计算，不走 AI
    stats = compute_stats(run_id)  # pass率、覆盖率、各团队对比

    # 层次 2：失败日志 MapReduce
    failed_logs = get_failed_logs(run_id)
    if len(failed_logs) > 500:
        batches = chunk(failed_logs, size=100)
        summaries = await asyncio.gather(*[
            summarize_batch(b, model="qwen-flash")  # 快模型 Map
            for b in batches
        ])
        text_insight = await reduce_summaries(summaries, model="qwen-plus")
    else:
        text_insight = await summarize_batch(failed_logs, model="qwen-plus")

    # 层次 3：全局报告
    report = await generate_report(
        structured_stats=stats,
        text_insight=text_insight,
        model="claude-sonnet"    # 最终报告用强模型
    )
    return report
💡 模型选型原则：快速分批摘要用 Haiku / Qwen-flash（便宜快），最终分析报告用 Sonnet / Qwen-plus（质量高）。

🔄 会话上限保护（防止 Context 无限膨胀）

单会话设置轮次上限，避免历史 Context 过长导致成本飙升和幻觉增加

🟡 优先级 P2 难度：低
问卷系统做法

单会话最多 50 轮，超出后新建会话，历史记录存 MongoDB 可追溯

测试平台应用

每个 Sprint 或每个 Story 开新会话，不同 Story 的测试用例生成不共享上下文

python
MAX_TURNS_PER_SESSION = 20   # 测试用例生成属于任务型，20轮足够

class TestCaseSession:
    def __init__(self, story_id: str):
        self.story_id = story_id
        self.conversation_id = generate_uuid()
        self.turns = []

    def add_turn(self, user_msg: str, ai_response: str):
        if len(self.turns) >= MAX_TURNS_PER_SESSION:
            # 超出限制：自动归档当前会话，提示用户开新会话
            self.archive()
            raise SessionLimitExceeded(
                f"会话已达 {MAX_TURNS_PER_SESSION} 轮上限，请开启新会话继续"
            )
        self.turns.append({"user": user_msg, "ai": ai_response})

    def get_context(self) -> list:
        # 只传最近 N 轮，而非全部历史
        return self.turns[-10:]  # 滑动窗口，控制 Token 消耗

📝 Prompt 云端版本管理 + 灰度切换

Prompt 变更可追踪、可回滚、支持 A/B 测试，保证可持续迭代

🟠 优先级 P1 难度：低
为什么必须版本管理

Prompt 是测试平台 AI 能力的核心资产。没有版本管理，每次"改一下 Prompt 试试"都是在赌博——不知道改好了还是改坏了，出问题也无法回滚。

实现方案 — GCP Secret Manager 存储版本
# Secret 命名规范：prompt-{场景}-v{版本号}
# prompt-testcase-gen-v1  →  当前生产版本
# prompt-testcase-gen-v2  →  灰度测试版本
# prompt-testcase-gen-v3  →  开发中版本

import os
from google.cloud import secretmanager

def get_prompt(scenario: str, team: str = "default") -> str:
    # 灰度路由：Checkout 团队使用 v2 测试
    version_map = {
        "checkout": "v2",   # 灰度
        "default": "v1",    # 生产稳定版
    }
    version = version_map.get(team, "v1")
    secret_name = f"prompt-{scenario}-{version}"

    client = secretmanager.SecretManagerServiceClient()
    name = f"projects/{PROJECT_ID}/secrets/{secret_name}/versions/latest"
    response = client.access_secret_version(request={"name": name})
    return response.payload.data.decode("UTF-8")

# 版本切换：不改代码，只更新 Secret
# gcloud secrets versions add prompt-testcase-gen-v2 --data-file=new_prompt.txt

🧪 AI 输出的自动化回归测试

每次改 Prompt 或换模型后，自动校验输出质量，防止引入回归风险

🟠 优先级 P1 难度：中
💡 测试平台用 AI 生成测试用例，本身也需要被测试。用测试的方式管理 AI，形成完整闭环。
python — AI 输出质量校验脚本
import pytest, json

class TestAITestCaseGeneration:
    """每次 Prompt 变更后自动运行此套件"""

    SAMPLE_STORIES = [
        {"title": "用户添加商品到购物车", "team": "checkout"},
        {"title": "库存扣减并发控制",    "team": "inventory"},
        {"title": "用户注册并登录",        "team": "customer"},
    ]

    def test_output_is_valid_json(self, story):
        result = call_ai_generate(story)
        parsed = json.loads(result)          # 必须可解析
        assert "test_cases" in parsed

    def test_has_all_scenario_types(self, story):
        result = call_ai_generate(story)
        types = {tc["type"] for tc in result["test_cases"]}
        assert "positive" in types     # 必须有正向场景
        assert "negative" in types     # 必须有负向场景
        assert "boundary" in types     # 必须有边界场景

    def test_no_duplicate_titles(self, story):
        result = call_ai_generate(story)
        titles = [tc["title"] for tc in result["test_cases"]]
        assert len(titles) == len(set(titles))  # 无重复标题

    def test_priority_is_valid_enum(self, story):
        result = call_ai_generate(story)
        valid = {"high", "medium", "low"}
        for tc in result["test_cases"]:
            assert tc["priority"] in valid  # 枚举合法

    def test_steps_not_empty(self, story):
        result = call_ai_generate(story)
        for tc in result["test_cases"]:
            assert len(tc["steps"]) >= 2    # 步骤不能为空

    def test_minimum_test_case_count(self, story):
        result = call_ai_generate(story)
        assert len(result["test_cases"]) >= 3  # 至少 3 个用例

在 GitHub Actions 中，每次 Prompt 变更的 PR 都自动触发此套件，合并前必须全部通过。

🛡️ 发送 AI 前数据脱敏

识别并过滤 Story/日志中的敏感信息，在"可分析"与"可合规"之间守住底线

🔴 优先级 P0（必做） 难度：低
⚠️ 中国区业务尤其需要注意：订单号、手机号、用户 ID、身份证号都属于敏感数据，发送给第三方 AI 平台前必须脱敏。
python — 脱敏处理
import re

SENSITIVE_PATTERNS = {
    "phone":    (r'1[3-9]\d{9}',            "[PHONE]"),
    "email":    (r'[\w.-]+@[\w.-]+\.\w+',    "[EMAIL]"),
    "id_card":  (r'\d{17}[\dXx]',            "[ID_CARD]"),
    "order_id": (r'ORD-\d{10,}',             "[ORDER_ID]"),
    "user_id":  (r'uid[=:]\s*\d+',           "[USER_ID]"),
}

def desensitize(text: str) -> str:
    """发送给 AI 前调用此函数"""
    for pattern, replacement in SENSITIVE_PATTERNS.values():
        text = re.sub(pattern, replacement, text)
    return text

# 使用
story_content = fetch_jira_story(story_id)
safe_content = desensitize(story_content)   # ← 脱敏后再发 AI
test_cases = await call_llm(safe_content)

AI 能力本身很重要，但把它工程化、可控化、可迭代更重要。
不追"最火技术名词"，围绕测试平台业务，把稳定性、准确性、合规性一项项打磨出来。
